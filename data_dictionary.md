# Data Dictionary — Bluestock MF Capstone
Database: `data/db/bluestock_mf.db` (SQLite)
Schema definition: `sql/schema.sql`

This document describes every table and column in the final database, including data
types, business meaning, and where the data originally came from.

---

## 1. dim_fund (Dimension Table)
One row per mutual fund scheme. 40 rows.
**Source:** `01_fund_master.csv` (AMFI India scheme master data)

| Column | Type | Description |
|---|---|---|
| amfi_code | TEXT (PK) | Unique AMFI scheme identifier. Sequentially assigned by AMFI; no embedded meaning beyond uniqueness. |
| fund_house | TEXT | Asset Management Company (AMC) that manages this scheme, e.g. "SBI Mutual Fund". |
| scheme_name | TEXT | Full official AMFI scheme name, including plan type (Direct/Regular) and option (Growth). |
| category | TEXT | Broad asset class: Equity or Debt (Hybrid is defined in the project spec but not present in this dataset). |
| sub_category | TEXT | Fund style within category, e.g. Large Cap, Small Cap, Liquid, Gilt, ELSS. |
| plan | TEXT | "Regular" (sold via distributor, includes commission) or "Direct" (bought directly, lower expense ratio). |
| launch_date | DATE | Date the scheme was launched. |
| benchmark | TEXT | The market index this fund is measured against (e.g. Nifty 100). |
| expense_ratio_pct | REAL | Annual fee charged by the fund, as % of assets. Validated range: 0.1%-2.5%. |
| exit_load_pct | REAL | Penalty % charged if redeemed before a minimum holding period. |
| min_sip_amount | REAL | Minimum monthly SIP investment allowed, in INR. |
| min_lumpsum_amount | REAL | Minimum one-time investment allowed, in INR. |
| fund_manager | TEXT | Name of the primary fund manager. |
| risk_category | TEXT | SEBI-defined risk tier. Observed values: Low, Moderate, Moderately High, High, Very High. |
| sebi_category_code | TEXT | SEBI's internal scheme classification code, e.g. EC01 = Large Cap Equity. |

## 2. dim_date (Dimension Table)
One row per calendar day from 2012-01-01 to 2026-12-31. 5,479 rows.
**Source:** Generated programmatically in `load_to_sql.py` (not from a provided CSV) —
needed as a calendar reference table to support date-based grouping (year/month/quarter)
without recalculating it in every query.

| Column | Type | Description |
|---|---|---|
| date_id | INTEGER (PK) | Surrogate key in YYYYMMDD integer format, e.g. 20220103. |
| full_date | DATE | The actual calendar date. |
| year | INTEGER | Calendar year. |
| month | INTEGER | Calendar month (1-12). |
| quarter | INTEGER | Calendar quarter (1-4). |
| day_of_week | TEXT | Day name, e.g. "Monday". |
| is_weekday | INTEGER | 1 if Mon-Fri, 0 if Sat/Sun. |

## 3. fact_nav (Fact Table)
One row per (fund, date) — daily NAV. 46,000 rows.
**Source:** `02_nav_history.csv`, cleaned in `data_cleaning.py`.

| Column | Type | Description |
|---|---|---|
| amfi_code | TEXT (FK -> dim_fund) | The fund this NAV belongs to. |
| date | DATE (part of composite PK) | Trading date. Business days only — no weekend/holiday rows exist in this dataset by design. |
| date_id | INTEGER (FK -> dim_date) | YYYYMMDD version of `date`, for joining to dim_date. |
| nav | REAL | Net Asset Value in INR on this date. Validated > 0; invalid values were forward-filled per fund during cleaning. |
| daily_return_pct | REAL | Day-over-day % change in NAV, computed during cleaning as (NAV_today / NAV_yesterday - 1) * 100, per fund. |

## 4. fact_transactions (Fact Table)
One row per investor transaction. 32,778 rows.
**Source:** `08_investor_transactions.csv`, cleaned in `data_cleaning.py`.

| Column | Type | Description |
|---|---|---|
| tx_id | INTEGER (PK, auto-increment) | Internal unique transaction ID (not present in source data; generated by SQLite). |
| investor_id | TEXT | Unique investor identifier (e.g. INV000001). |
| amfi_code | TEXT (FK -> dim_fund) | The fund involved in this transaction. |
| transaction_date | DATE | Date of the transaction. |
| date_id | INTEGER (FK -> dim_date) | YYYYMMDD version of transaction_date. |
| transaction_type | TEXT | One of: SIP, Lumpsum, Redemption. Validated against this enum during cleaning. |
| amount_inr | REAL | Transaction amount in INR. Validated > 0. |
| state | TEXT | Investor's Indian state (12 states covered). |
| city | TEXT | Investor's city. |
| city_tier | TEXT | T30 (Top 30 cities) or B30 (Beyond Top 30), per AMFI's official classification. |
| age_group | TEXT | Investor age bracket, e.g. 26-35. |
| gender | TEXT | Male / Female. |
| annual_income_lakh | REAL | Investor's annual income, in INR lakh. |
| payment_mode | TEXT | UPI / Net Banking / Mandate / Cheque. |
| kyc_status | TEXT | Verified or Pending. Validated against this enum during cleaning. |

## 5. fact_performance (Fact Table)
One row per fund — computed performance and risk metrics. 40 rows.
**Source:** `07_scheme_performance.csv`, cleaned in `data_cleaning.py`. Identifying columns
(scheme_name, fund_house, category, plan) were excluded from this table since they already
exist in `dim_fund` — joining on amfi_code retrieves them without duplication.

| Column | Type | Description |
|---|---|---|
| amfi_code | TEXT (PK, FK -> dim_fund) | The fund these metrics belong to. |
| return_1yr_pct | REAL | 1-year absolute return %. |
| return_3yr_pct | REAL | 3-year CAGR %. |
| return_5yr_pct | REAL | 5-year CAGR %. |
| benchmark_3yr_pct | REAL | The fund's benchmark index 3-year CAGR, for comparison. |
| alpha | REAL | Return above benchmark (return_3yr_pct - benchmark_3yr_pct). |
| beta | REAL | Sensitivity to overall market movement (1.0 = moves with the market). |
| sharpe_ratio | REAL | Risk-adjusted return using total volatility. Higher is better; >1 is considered good. Note: Liquid Fund schemes show unusually high values due to very low volatility in the denominator — see notes in reports/day1_data_quality_notes.md. |
| sortino_ratio | REAL | Like Sharpe, but only penalises downside volatility. Same Liquid Fund caveat applies. |
| std_dev_ann_pct | REAL | Annualised standard deviation of daily returns — a measure of volatility/risk. |
| max_drawdown_pct | REAL | Worst peak-to-trough decline, always <= 0 by definition. |
| aum_crore | REAL | Scheme-level Assets Under Management, in INR crore (distinct from fund-house-level AUM in fact_aum). |
| expense_ratio_pct | REAL | Duplicated from dim_fund for convenience when querying performance directly. |
| morningstar_rating | INTEGER | 1-5 star rating (simulated, based on Sharpe ratio). |
| risk_grade | TEXT | Risk classification associated with this fund's performance profile. |

## 6. fact_aum (Fact Table)
One row per (fund house, date) — quarterly AUM by fund house. 90 rows.
**Source:** `03_aum_by_fund_house.csv`, cleaned in `data_cleaning.py`.

| Column | Type | Description |
|---|---|---|
| fund_house | TEXT (part of composite PK) | The AMC this AUM figure belongs to. Note: this is fund-house-level data, not scheme-level (unlike fact_performance.aum_crore). |
| date | DATE (part of composite PK) | Quarter-end reporting date. |
| date_id | INTEGER (FK -> dim_date) | YYYYMMDD version of date. |
| aum_crore | REAL | Total AUM for this fund house, in INR crore. |
| aum_lakh_crore | REAL | Same figure expressed in INR lakh crore (industry-standard reporting unit for large totals). 1 lakh crore = 100,000 crore. |
| num_schemes | INTEGER | Number of schemes this fund house operates, as of this date. |

---

## 7. fact_sip_industry (Fact Table)
One row per month — industry-wide SIP statistics. 48 rows.
**Source:** `04_monthly_sip_inflows.csv`, cleaned in `data_cleaning.py`.

| Column | Type | Description |
|---|---|---|
| month | TEXT (PK) | Format YYYY-MM. |
| sip_inflow_crore | REAL | Total SIP inflows across the industry that month, in INR crore. |
| active_sip_accounts_crore | REAL | Number of actively contributing SIP accounts, in crore. |
| new_sip_accounts_lakh | REAL | New SIP registrations that month, in lakh accounts. |
| sip_aum_lakh_crore | REAL | Total SIP-related AUM, in INR lakh crore. |
| yoy_growth_pct | REAL | Year-over-year growth % in SIP inflows. NULL for the first 12 months (Jan-Dec 2022), since no prior-year data exists to compare against — this is expected, not missing data. |
